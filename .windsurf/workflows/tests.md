# مسار عمل فحص الجودة والاختبارات المتقدمة (Quality Control & Integration/E2E Pipeline)

## 🛑 أولاً: المعايير الهندسية الصارمة (Strict Engineering Guidelines)

يجب عليك الالتزام الصارم بالمعايير التالية أثناء كتابة أي كود أو إعداد بيئة الاختبار. لا يُسمح بأي استثناءات:

- **المنهجية والهدف (Case Closed)**: الحل البرمجي يجب أن يكون نهائياً، شاملاً، وجاهزاً للتشغيل الفوري (Production-ready). لا تقدم أي أنصاف حلول، ولا تترك تعليقات مثل `// Add logic here`، ولا تستخدم أكواد وهمية.
- **البنية البرمجية (Architecture)**: اعتمد على البرمجة كائنية التوجه (OOP) مع البرمجة الوظيفية (Functional Programming). يجب أن يكون الكود معيارياً ومقسماً (Modular Design) في ملفات منفصلة. يُمنع منعاً باتاً وضع الكود بالكامل في ملف واحد.
- **واقعية الاختبارات (Real Integration - ZERO MOCKS)**: قم ببرمجة اختبارات التكامل لتتصل بقواعد بيانات حقيقية (مخصصة للاختبار) وواجهات برمجة تطبيقات (APIs) فعلية. يُمنع نهائياً وبشكل قاطع استخدام البيانات الوهمية (Mocks) أو (Stubs).
- **الأداء والتزامن**: استخدم `async/await` في كافة عمليات الإدخال والإخراج (I/O)، استدعاءات الشبكة، والتعامل مع قواعد البيانات.
- **الصلابة (Robustness) ومعالجة الأخطاء**:
  - استخدم كتل `try/catch` في كافة العمليات المعرضة للأخطاء.
  - 🚫 **قانون صارم**: يُمنع منعاً باتاً استخدام `console.log` تحت أي ظرف وفي أي ملف.
  - قم بإعداد نظام تسجيل احترافي (Professional Logging) باستخدام مكتبة `pino` (أسرع وأقل استهلاكاً للموارد) أو `winston` لتتبع مسار الاختبارات ونتائجها.
- **إدارة الإعدادات (Config Management)**: قم بإنشاء فئة `ConfigManager` تعتمد على بيئة التشغيل (Environment Variables) وتستخدم أداة `zod` مع `dotenv` للتحقق من صحة أنواع البيانات (Type Validation) لمسارات قواعد البيانات وبيانات الاعتماد، لضمان أمان واستقلالية بيئة الاختبار.
- **التوثيق الداخلي**: اكتب توثيقاً داخلياً (Comments) باللغة العربية فقط داخل الكود نفسه لشرح الغرض من كل مكون، ولا تقدم شروحات نصية خارج الكود في ردودك إلا إذا طُلب منك ذلك صراحة؛ دع الكود يتحدث عن نفسه.

## 🛠️ ثانياً: الأدوات وإدارة التبعيات (Dependency Management)

تأكد من استخدام الإصدارات الأدق والأكثر استقراراً وتوافقاً مع بيئة TypeScript الحديثة:

- **E2E Testing**: حزمة `@playwright/test` (لدعم TS الأصلي والتعامل مع المتصفحات الحديثة).
- **Integration Testing**: حزمة `supertest` لاختبار نقاط النهاية، مع إطار العمل `vitest` (موصى به لسرعته وتوافقه المدمج مع TS كبديل أسرع لـ Jest).
- **Config & Validation**: حزم `zod` و `dotenv`.
- **Logging**: حزم `pino` أو `winston`.

## 🔄 ثالثاً: خطوات مسار العمل الإلزامية (Sequential Pipeline Steps)

قم بتنفيذ الخطوات التالية بالترتيب. في حال فشل أي خطوة، قم بتنفيذ "إصلاح الأخطاء (Auto-Fix)" فوراً ولا تنتقل للخطوة التالية إلا بعد نجاح الحالية بالكامل.

### الخطوة 1: تهيئة البيئة وتثبيت الاعتماديات

- **الإجراء**: إعداد بيئة العمل باستخدام Node.js 20 و pnpm 9.
- **التنفيذ**: تشغيل الأمر `pnpm install --frozen-lockfile`. (تأكد من تثبيت أدوات الاختبار والـ Logging المطلوبة).
- **إصلاح الأخطاء**: في حالة فشل التثبيت، ابحث فوراً عن تعارض الاعتماديات أو مشاكل ملف القفل (`pnpm-lock.yaml`)، وقم بتحديث الحزم المتعارضة بأكثر الإصدارات استقراراً، ثم أعد التثبيت.

### الخطوة 2: فحص وتصحيح التنسيق (Formatting)

- **الإجراء**: التأكد من التزام الكود بمعايير التنسيق.
- **التنفيذ**: تشغيل الأمر `pnpm run format:check`.
- **إصلاح الأخطاء**: إذا فشل الفحص، قم بتشغيل `pnpm run format` للإصلاح التلقائي. قم بتعديل ملفات الإعدادات أو الكود يدوياً لحل أي تعارض مستمر (مثل إعدادات Prettier).

### الخطوة 3: فحص الأخطاء المنطقية (Linting)

- **الإجراء**: تحليل الكود لاكتشاف الأخطاء المنطقية ومخالفات معايير الكتابة.
- **التنفيذ**: تشغيل الأمر `pnpm run lint`.
- **إصلاح الأخطاء**: إذا فشل، قم بتشغيل `pnpm run lint --fix`. تدخل برمجياً لكتابة الحلول الجذرية لأي أخطاء Linting متبقية وعدّل الكود مباشرة ليتوافق مع القواعد.

### الخطوة 4: فحص الأنواع (Type Checking)

- **الإجراء**: التحقق من صحة أنواع البيانات (TypeScript).
- **التنفيذ**: تشغيل الأمر `pnpm run typecheck`.
- **إصلاح الأخطاء**: في حالة ظهور أخطاء، حدد الملفات المتضررة واكتب التعديلات لضبط تعريفات الأنواع (Interfaces/Types) لضمان توافقها التام دون استخدام حلول مؤقتة مثل `any` أو `@ts-ignore`.

### الخطوة 5: الاختبارات الوظيفية (Unit Testing)

- **الإجراء**: التأكد من سلامة منطق العمليات البرمجية المعزولة.
- **التنفيذ**: تشغيل الأمر `pnpm run test` (باستخدام Vitest).
- **إصلاح الأخطاء**: عند فشل أي اختبار، افحص كود الاختبار والكود المصدري المرتبط به. حدد المشكلة الهندسية الأساسية، واكتب الكود اللازم لإصلاح الخلل الوظيفي.

### الخطوة 6: اختبارات التكامل الحقيقي (Real Integration Testing)

- **الإجراء**: التحقق من تكامل نقاط النهاية (Endpoints) والاتصال بالخدمات وقواعد البيانات الفعلية.
- **التنفيذ**: كتابة وتشغيل الاختبارات باستخدام `supertest` مع `vitest`.
- **إصلاح الأخطاء**: إذا فشل الاتصال بقاعدة البيانات أو تم إرجاع استجابات غير متوقعة، راجع فئة `ConfigManager`، وقم بتصحيح الاتصال أو معالجة الخلل في الكود المصدري لضمان التكامل الفعلي.

### الخطوة 7: اختبارات من البداية للنهاية (End-to-End Testing)

- **الإجراء**: محاكاة المستخدم الفعلي وتفاعلات واجهة المستخدم للمسارات الحرجة في المشروع.
- **التنفيذ**: كتابة وتشغيل الاختبارات باستخدام `@playwright/test`.
- **إصلاح الأخطاء**: في حال فشل اختبارات مسار المستخدم، قم بفحص محددات العناصر (Selectors) أو مشكلات التزامن (Timeouts/Async)، وتعديل كود الاختبار أو كود الواجهة الأمامية ليجتاز الاختبارات.

### الخطوة 8: بناء المشروع (Build)

- **الإجراء**: تجميع وبناء المشروع لبيئة الإنتاج.
- **التنفيذ**: تشغيل الأمر `pnpm run build`.
- **إصلاح الأخطاء**: إذا فشلت عملية البناء، قم بتحليل أخطاء التجميع (Bundling)، أو الاستيراد المفقود، أو أخطاء الذاكرة. قم بتعديل إعدادات البناء أو هيكلة الملفات لضمان اكتمال عملية البناء بنجاح.

### الخطوة 9: اختبار الدخان (Smoke Test)

- **الإجراء**: التحقق من وجود مخرجات البناء وسلامة ملفات بدء التشغيل.
- **التنفيذ**: فحص مسارات الخرج البرمجي للتحقق من وجود مجلدات الإنتاج (`dist`، `.next`، أو `build`)، والتحقق من وجود ملف المدخل الرئيسي (مثل `index.html`).
- **إصلاح الأخطاء**: إذا لم يتم العثور على المخرجات المطلوبة، راجع ملفات إعداد البناء (مثل `vite.config.ts` أو `next.config.js`)، وحدد سبب عدم توليد الملفات، واكتب التعديل لتصحيح المسارات وإعادة البناء.

### الخطوة 10: التحقق النهائي (Final Validation)

- **الإجراء**: تنفيذ المراجعة الشاملة الأخيرة.
- **التنفيذ**: تشغيل الأمر `pnpm run validate`.
- **إصلاح الأخطاء**: معالجة أي قصور أخير يظهر في هذه المرحلة برمجياً بشكل كامل لتسليم كود جاهز تماماً للإنتاج (Production-ready).

## 🚀 رابعاً: الإجراء الفوري المطلوب (Immediate Action Trigger)

قم بتحليل الملفات المرتبطة بالوظيفة المطلوبة، وابدأ فوراً في كتابة بيئة الاختبار وهيكلتها بناءً على هذه الشروط. ابدأ بإنشاء `ConfigManager`، وإعداد نظام التسجيل (Logger)، وكتابة اختبارات التكامل والـ E2E.

**تذكر**: أريد الكود مقسماً في ملفات منفصلة مع التعليقات العربية من الداخل فقط، ولا تقدم أي شروحات نصية خارج الكود. انطلق.
